    Security

    GoBlocks follows WordPress security best practices throughout: every input is sanitized, every output is escaped, every state-modifying request is verified with a nonce or capability check, and no database queries use raw user input.

    Input Sanitization

    Block Attributes

    Block attributes are validated and sanitized by WordPress's block parser against the JSON schema in each block's block.json file. GoBlocks defines strict attribute types:

    {
  "attributes": {
    "padding": { "type": "integer", "minimum": 0, "maximum": 500 },
    "backgroundColor": { "type": "string", "pattern": "^#[0-9a-fA-F]{3,8}$" },
    "tag": { "type": "string", "enum": ["div","section","article","aside","header","footer","main"] }
  }
}

    Attributes that do not match the schema are rejected by the WordPress block parser before the render callback is reached. No raw attribute value is ever used without prior schema validation.

    Settings

    All values submitted to GoBlocks → Settings are validated against a JSON schema before saving:

    • container_width — sanitized with absint(), clamped to 320–3840
    • css_print_method — validated against allowlist ['file', 'inline']
    • breakpoints — each value sanitized with absint(), validated as ≥ 1
    • global_color_palette — each color value matched against /^#[0-9a-fA-F]{3,8}$/

    Dynamic Content Tags

    The dynamic tag parser uses a strict allowlist: only registered tag slugs are resolved. An unrecognized tag slug is returned as an empty string — it is never executed as code. Tag option values are sanitized at resolution time using the appropriate function for the output context.

    Output Escaping

    GoBlocks escapes all output at the point of output, using the function matched to the context:

    ContextFunction used
    HTML text contentesc_html()
    HTML attribute valuesesc_attr()
    URLs (href, src)esc_url()
    Rich HTML content (allowed tags)wp_kses_post()
    CSS valuessanitize_hex_color() / absint() / value allowlist
    JavaScript (inline data)wp_json_encode()

    Dynamic tag output is escaped with esc_html() by default. Custom tags that return HTML should escape their own output and not rely on GoBlocks's default escaping.

    Capability Checks

    All administrative actions require the manage_options capability (WordPress Administrator role):

    • Saving settings via the admin UI
    • Clearing the CSS cache
    • All REST API endpoints under /wp-json/goblocks/v1/
    • WP-CLI commands that modify data
    <?php
// REST API endpoint permission callback
public function permissions_check(): bool {
    return current_user_can( 'manage_options' );
}

    Nonce Verification

    All admin form submissions and Ajax requests include a nonce:

    • Admin settings form: Verified with check_admin_referer( 'goblocks_settings' )
    • REST API: Authenticated via the wp_rest nonce in the request header (X-WP-Nonce), which WordPress validates automatically
    • Ajax actions: Verified with check_ajax_referer( 'goblocks_ajax' )

    CSS File Security

    Generated CSS files are stored in wp-content/uploads/goblocks/. GoBlocks protects this directory:

    • An index.php file with a blank body is placed in the directory to prevent directory listing.
    • CSS filenames are based on post IDs (e.g. 42.css) — they do not contain user-supplied data and cannot be exploited for path traversal.
    • The CSS content is generated entirely from validated block attributes — no user-supplied content is written directly to the file.

    SQL Injection Prevention

    GoBlocks does not construct raw SQL queries. All database interactions use WordPress's $wpdb methods with prepared statements where custom queries are needed:

    <?php
$result = $wpdb->get_results(
    $wpdb->prepare(
        'SELECT post_id FROM %i WHERE option_name = %s',
        $table_name,
        $option_name
    )
);

    Query block arguments are filtered through WP_Query's built-in argument sanitization. The goblocks_query_args filter exposes the final WP_Query args array — custom filter callbacks that add raw user input to this array should use absint() or explicit allowlists for any untrusted values.

    XSS Prevention

    GoBlocks's block renderer never outputs unescaped HTML from user-supplied attributes. SVG output from the Icon block is generated entirely from the registered icon library (a server-side array of allowed SVG paths) — user input never reaches SVG path data.

    The Icon block's 112 icons are registered as a PHP array of sanitized SVG strings. The block attribute only stores the icon slug — the slug is matched against the allowlist, and the pre-sanitized SVG is output. An unrecognized slug outputs nothing.

    Reporting a Vulnerability

    If you discover a security vulnerability in GoBlocks, please do not report it publicly. Instead:

    1. Email security@godevs.net with a description of the vulnerability, steps to reproduce, and the affected version.
    2. You will receive a response within 48 hours acknowledging the report.
    3. We aim to release a patch within 7 days of confirmation for critical vulnerabilities.

    We follow responsible disclosure and will credit researchers in the release notes (with your permission).
