    Accessibility

    GoBlocks is built to comply with WCAG 2.1 Level AA. Every interactive block implements the relevant ARIA Authoring Practices Guide pattern. This page documents the ARIA semantics, keyboard support, and focus management for each interactive block.

    General Practices

    • All blocks use semantic HTML5 elements as the primary structure before falling back to ARIA roles.
    • Every focusable element has a visible focus indicator that meets the WCAG 2.1 AA focus-visible requirement.
    • Color is never the sole means of conveying information — icons, labels, or patterns supplement color-coded states.
    • All image blocks output alt text (either from the media library metadata or the author-supplied field). Empty alt (alt="") is used for purely decorative images.
    • GoBlocks honors prefers-reduced-motion: all CSS transitions and JavaScript animations respect this media query and are disabled or shortened when the user prefers reduced motion.

    Tabs Block

    Implements the ARIA Tabs Pattern.

    ElementRole / Attribute
    Tab containerrole="tablist", aria-label
    Each tab buttonrole="tab", aria-selected="true/false", aria-controls="{panel-id}", id="{tab-id}"
    Each panelrole="tabpanel", aria-labelledby="{tab-id}", id="{panel-id}"

    Keyboard support:

    • ← / → — move focus between tabs (horizontal tablist)
    • ↑ / ↓ — move focus between tabs (vertical tablist)
    • Home — move focus to the first tab
    • End — move focus to the last tab
    • Space / Enter — activate the focused tab

    Accordion Block

    Built on native <details>/<summary> HTML elements — keyboard and screen reader support is provided natively by the browser.

    ElementHTML Element
    Accordion item<details>
    Trigger (question)<summary>
    Content (answer)<div> inside <details>

    No additional ARIA is required. Enter or Space on <summary> toggles the disclosure.

    When FAQ Schema is enabled, the Accordion outputs itemscope and itemtype="https://schema.org/Question" / Answer attributes without affecting the HTML semantics used by assistive technologies.

    Modal Block

    Implements the ARIA Dialog Pattern.

    ElementRole / Attribute
    Dialog containerrole="dialog", aria-modal="true", aria-labelledby="{title-id}"
    Trigger button<button> with descriptive label
    Close buttonaria-label="Close dialog"

    Focus management: When the modal opens, focus is moved to the first focusable element inside the dialog. Focus is trapped inside the dialog while it is open — Tab and Shift+Tab cycle only within the dialog. When the dialog closes, focus is returned to the trigger button.

    Keyboard support:

    • Escape — closes the dialog
    • Tab / Shift+Tab — cycles focus within the dialog

    Navigation Block

    ElementRole / Attribute
    Nav wrapper<nav aria-label="Primary navigation">
    Submenu togglearia-expanded="true/false", aria-controls="{submenu-id}"
    Submenu listid="{submenu-id}"
    Mobile hamburgeraria-label="Toggle menu", aria-expanded

    Submenus open on both click and keyboard (Enter or Space on the toggle button). Escape closes an open submenu and returns focus to the toggle.

    Slider Block

    ElementRole / Attribute
    Slider regionrole="region", aria-label="Slider"
    Slide listrole="tablist" (dot indicators)
    Dot indicatorrole="tab", aria-selected, aria-label="Slide N"
    Previous / Next buttonsaria-label="Previous slide" / "Next slide"

    Autoplay is paused when the slider receives keyboard focus or the user's mouse enters the slider, preventing users from missing content.

    Counter Block

    The animated number element is wrapped in a <span aria-live="polite">. Screen readers will announce the final value once the animation completes. The element is not announced during each frame of the animation to avoid overwhelming assistive technology users.

    Progress Bar Block

    <div
  role="progressbar"
  aria-valuenow="75"
  aria-valuemin="0"
  aria-valuemax="100"
  aria-label="Skill level: JavaScript"
>

    The aria-label is set from the label field in the block Inspector. Ensure the label is descriptive (e.g. "JavaScript proficiency: 75%") rather than generic ("Progress bar").

    Alert Block

    VariantARIA role
    Inforole="note"
    Successrole="status"
    Warningrole="note"
    Errorrole="alert"

    role="alert" causes screen readers to announce the content immediately when the element appears in the DOM. Use the Error variant only for critical messages. The Info and Warning variants use role="note", which is read in document order without interrupting the screen reader's current position.

    Video Block

    The lazy facade (poster image with a play button) is rendered as a <button> element with aria-label="Play video: {title}". This ensures the play trigger is keyboard-accessible and properly announced by screen readers.

    For self-hosted videos, GoBlocks passes through the <video> element's native browser controls, which are keyboard and screen reader accessible. For embedded videos (YouTube, Vimeo), the iframe is given a descriptive title attribute.

    Authoring Best Practices

    • Headings: Use the Heading block with the correct heading level for your document outline. Do not skip heading levels (h2 → h4) for visual appearance — use padding/margin instead.
    • Links vs. buttons: Use Button blocks with a valid href for navigation links. Use <button> (the default when href is empty) for actions that do not navigate.
    • Images: Always set meaningful alt text on Image blocks. For decorative images (backgrounds, illustration accents), use the Decorative image toggle to output alt="".
    • Color contrast: GoBlocks's default color palette meets WCAG AA 4.5:1 contrast for normal text and 3:1 for large text. Verify custom palette colors with a contrast checker before use.
    • Animation: Keep animations subtle and short. GoBlocks automatically disables block animations when prefers-reduced-motion is active.
