    Troubleshooting

    Before reporting a bug, please try the steps below. Most issues resolve with one of these solutions.

    Blocks Not Appearing in the Inserter

    Symptoms: The GoBlocks category is missing from the block inserter, or individual GoBlocks blocks do not appear.

    Causes and solutions:

    1. Plugin not activated: Go to Plugins → Installed Plugins and verify GoBlocks shows Active status.
    2. Classic Editor plugin active: The Classic Editor plugin disables the block editor entirely. Deactivate the Classic Editor plugin to use GoBlocks.
    3. PHP version too low: GoBlocks requires PHP 8.0+. Check your PHP version in Tools → Site Health → Info → Server. Upgrade PHP if it is below 8.0.
    4. Plugin conflict: Deactivate all other plugins except GoBlocks and test. Reactivate one by one to identify the conflicting plugin.
    5. Theme conflict: Switch to the Twenty Twenty-Four theme temporarily and test. If blocks appear, the issue is theme-specific.

    CSS Not Loading / Blocks Appear Unstyled

    Symptoms: GoBlocks blocks appear in the frontend but without styles — no colors, incorrect spacing, boxes without borders.

    Causes and solutions:

    1. Uploads directory not writable: GoBlocks needs write access to wp-content/uploads/goblocks/. Check permissions — the directory should be 755 or 775 and owned by the web server user. 
      ls -la wp-content/uploads/goblocks/
    2. CSS Print Method set to inline but something is stripping <style> tags: Check if a security plugin or server-level WAF is removing inline style tags.
    3. CSS file blocked by server: Try accessing the CSS file URL directly in your browser (e.g. /wp-content/uploads/goblocks/42.css). A 403 or 404 response indicates a server configuration issue.
    4. Page not yet saved: CSS is generated on save. Open the page in the editor and save it to trigger CSS generation.
    5. Switch to inline mode temporarily: Go to GoBlocks → Settings → CSS Print Method → set to inline → Save. If styles appear, the file writing or serving is the problem.

    Block Editor Crashes or Shows "Block has encountered an error"

    Symptoms: A GoBlocks block shows an error boundary in the editor with "Block has encountered an error" and an "Attempt Recovery" button.

    Causes and solutions:

    1. WordPress version too low: Update WordPress to 6.5 or newer. GoBlocks uses Block API v3 features not available in older versions.
    2. Stale browser cache: After a GoBlocks update, your browser may have cached the old block editor scripts. Hard-refresh with Ctrl+Shift+R (Windows) or Cmd+Shift+R (Mac).
    3. JavaScript conflict: Open your browser's developer tools → Console. Look for JavaScript errors. If an error mentions another plugin, that plugin may be conflicting. Deactivate other plugins one by one to isolate.
    4. Corrupted block markup: Use Attempt Recovery which will try to parse the block from its saved attributes. If recovery fails, you may need to edit the post in Code view and correct the block comment.

    Dynamic Content Tags Not Resolving

    Symptoms: {post_title} or other tags appear literally in the frontend HTML source.

    Causes and solutions:

    1. Used in an unsupported block or field: Dynamic tags only work in the blocks and fields listed in Dynamic Content → Supported Blocks. Tags in core WordPress blocks are not parsed.
    2. Used outside a Query Loop without post context: On archive or home pages, post-specific tags resolve to an empty string. Use fallback: {post_title|fallback:No title}.
    3. Full-page cache serving old content: If you recently added dynamic tags to an existing page, purge your full-page cache to regenerate the HTML.
    4. Tag syntax error: Check that the tag uses curly braces ({}) and not square brackets or other delimiters. Options must use a pipe: {post_date|format:Y}.

    Interactive Blocks Not Working on Frontend

    Symptoms: Tabs, Accordion, Modal, Slider, or Counter do not respond to user interaction on the frontend.

    Causes and solutions:

    1. JavaScript file not loaded: Open browser devtools → Network tab → search for goblocks. If the block's JS file is absent, GoBlocks may not be detecting the block with has_block().
      • Verify the block is present on the page (not inside a nested Query Loop on a template). Some server-side caches cache the page before GoBlocks can detect the block.
      • Try adding the block directly to a standard post or page (not a template) and testing there.
    2. JavaScript error from another plugin: Open browser devtools → Console. If there is a JS error from another plugin on the same line as GoBlocks initialization, that error may be preventing GoBlocks scripts from running. Deactivate plugins one by one.
    3. Optimization plugin deferring or merging GoBlocks JS: Plugins like WP Rocket's JS optimization can break GoBlocks's event listeners by deferring scripts. Exclude GoBlocks JS files from deferral:
    wp-content/plugins/goblocks/assets/js/

    Patterns Not Appearing in the Pattern Library

    Symptoms: GoBlocks patterns are missing from the Patterns tab in the block inserter.

    Causes and solutions:

    1. Pattern category not registered: GoBlocks registers its pattern category on the init hook. If a conflict occurs during init, the category may not be registered. Check the PHP error log for fatal errors.
    2. WordPress version below 6.0: The block pattern API used by GoBlocks requires WordPress 6.0+. Update WordPress.
    3. Block pattern caching: In some configurations, WordPress caches the pattern registry. Deactivate any pattern-caching plugins and test.

    CSS Regenerating on Every Page Load

    Symptoms: Every frontend page load takes slightly longer than expected. Checking the uploads directory shows the CSS file's timestamp changing on every request.

    Causes and solutions:

    1. Object cache not persisting post meta: GoBlocks uses WordPress's object cache to store the block fingerprint (hash). If your site does not have a persistent object cache (Redis or Memcached), the hash is re-computed on every request. Install a persistent object cache plugin (Redis Object Cache is recommended for most hosts).
    2. File system permissions preventing the CSS file from being read: GoBlocks checks whether the existing CSS file is up-to-date by reading its modification time. If the file cannot be read (due to permissions), GoBlocks regenerates it. Check permissions on wp-content/uploads/goblocks/.

    WP-CLI Commands Fail

    Symptoms: Running wp goblocks cache clear or GoBlocks WP-CLI commands returns an error.

    Causes and solutions:

    1. WP-CLI version too low: GoBlocks requires WP-CLI 2.7+. Run wp cli version to check. Update with wp cli update.
    2. WordPress not fully loaded in CLI context: Some WP-CLI commands run without the full WordPress initialization. GoBlocks CLI commands require the full WordPress environment. Use wp --require=wp-load.php goblocks … if commands fail to bootstrap.
    3. Missing capability: WP-CLI commands that modify GoBlocks settings run as the configured CLI user. Ensure the CLI user has the manage_options capability.

    Still Stuck?

    If none of the above solutions resolve your issue:

    1. Check the WordPress.org support forum for similar reports.
    2. Enable WP_DEBUG in wp-config.php and check wp-content/debug.log for PHP errors related to GoBlocks.
    3. Open a new thread in the support forum with your WordPress version, PHP version, list of active plugins, and a description of the issue including any error messages.

    See Support for full contact information and response time expectations.
