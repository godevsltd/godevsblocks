    Performance

    GoBlocks is engineered to have minimal impact on page-load performance. This page explains how CSS delivery, JavaScript loading, and browser caching work.

    Per-Page Static CSS Files

    GoBlocks does not load a single global stylesheet containing CSS for all 36 blocks. Instead, it generates a per-page CSS file that contains only the styles needed by the blocks on that specific page.

    Flow:

    1. A post or page is saved in the block editor.
    2. GoBlocks traverses the parsed block tree and collects CSS declarations from each GoBlocks block's attributes.
    3. A CSS string is assembled and written to wp-content/uploads/goblocks/{post-id}.css.
    4. On the frontend, that file is enqueued with a <link> tag. Only that one file loads.

    Benefits:

    • A page with 5 blocks loads a CSS file of a few kilobytes — not a 100 KB global stylesheet.
    • The CSS file is a static file served directly by your web server with browser cache headers — no PHP or WordPress needed for repeat visitors.
    • The file is served from /wp-content/uploads/, which is typically on a CDN or object storage when using managed WordPress hosts.

    Delta Regeneration

    CSS files are not regenerated on every save. GoBlocks uses block fingerprinting: before writing a new file, it computes a hash of the block attributes and compares it to the stored hash. If the hash has not changed, the existing CSS file is kept and no write occurs.

    A CSS file is only regenerated when:

    • The post is saved with block attribute changes that affect CSS output.
    • GoBlocks settings that affect CSS (container width, breakpoints, global color palette) are updated.
    • The CSS cache is manually cleared via GoBlocks → Settings → Clear CSS Cache or the REST API.

    Result: Even on pages that are frequently updated, the CSS file is regenerated only when the visual design changes — not on every text edit.

    Browser Cache Busting

    The CSS file is enqueued with a version query string derived from the file's last-modified timestamp:

    <link rel="stylesheet" href="/wp-content/uploads/goblocks/42.css?v=1704067200">

    When the CSS is regenerated (because the page design changed), the timestamp changes, which causes browsers to fetch the updated file instead of serving the stale cached version. Unchanged pages always serve the cached version.

    Inline CSS Fallback

    When the CSS print method is set to 'inline' (or when the uploads directory is not writable), GoBlocks injects styles directly into <style> tags in the page <head>. Inline CSS cannot be browser-cached across pages, but it eliminates the extra HTTP request for the CSS file — useful when the page is already cached at the full-page level by a caching plugin.

    To switch between modes: GoBlocks → Settings → CSS Print Method.

    Conditional JavaScript Loading

    GoBlocks block JavaScript is enqueued only when the block is present on the page. This is implemented using WordPress's has_block() function.

    <?php
// Only enqueue slider.js if the Slider block is on the page.
if ( has_block( 'goblocks/slider' ) ) {
    wp_enqueue_script( 'goblocks-slider', ... );
}

    Core Web Vitals Strategy

    GoBlocks is designed with Core Web Vitals in mind:

    MetricGoBlocks technique
    LCP (Largest Contentful Paint) Per-page CSS is render-blocking but tiny (typically 2–8 KB). Hero images can use fetchpriority="high" via the Image block's priority setting. Video blocks use a lazy facade so the iframe does not block initial render.
    CLS (Cumulative Layout Shift) Image blocks always output explicit width and height attributes (from media library metadata). Slider, Tabs, and Accordion do not cause layout shift on load — they render their initial state without JavaScript transitions.
    INP (Interaction to Next Paint) All interactive block scripts are minimal, synchronous-free, and event-delegation based. The heaviest block JS (Slider at ~3 KB) executes in under 1 ms on a mid-range mobile device.

    Compatibility with Caching Plugins

    GoBlocks is compatible with all major WordPress caching plugins. Recommended configuration:

    PluginRecommendation
    WP RocketExclude /wp-content/uploads/goblocks/ from minification. Enable CDN for this path.
    W3 Total CacheNo special configuration required. GoBlocks CSS files are detected as static assets.
    WP Super CacheNo special configuration required.
    LiteSpeed CacheNo special configuration required. GoBlocks CSS is served as a static file.
    CloudflareGoBlocks CSS files are automatically cached at the CDN edge. Cache is busted by the version query string when CSS regenerates.

    Caution: Some aggressive minification plugins may attempt to inline or merge GoBlocks CSS files. If you notice styling issues after enabling a new caching/optimization plugin, exclude /wp-content/uploads/goblocks/*.css from that plugin's CSS processing.

    Clearing the CSS Cache

    Via admin UI: GoBlocks → Settings → Clear CSS Cache.

    Via WP-CLI:

    # Clear all CSS cache files
wp eval 'GoBlocks\CSS\CssEnqueue::clear_all_cache();'

# Clear cache for a specific post
wp eval 'GoBlocks\CSS\CssEnqueue::clear_post_cache( 42 );'

    Via REST API:

    DELETE /wp-json/goblocks/v1/css/cache
DELETE /wp-json/goblocks/v1/css/cache/42

    After clearing the cache, each page's CSS will be regenerated on the next frontend request. On high-traffic sites, consider warming the cache by crawling the site after clearing.
