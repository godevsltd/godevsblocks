Market Context

The Problem with Existing Block Plugins

Architectural Debt

  • Flat triplets — widthMobile, widthTablet
  • Inline style="" on block elements — security & specificity debt
  • PHP regenerates CSS at render time — server overhead
  • Plain JS — no TypeScript, no type safety

WP.org Quality Issues

  • Plugin Check errors at first submission
  • !important overuse, unsafe eval, global leaks
  • Zero escaping at output — XSS vectors

What the Market Needs

  • One CSS engine — TypeScript generates, PHP only caches
  • Nested responsive { base, sm, md, lg } values
  • TypeScript strict — zero any
  • PHPStan Level 6 + WPCS — zero violations
  • Plugin Check: 0 errors before first submission

The GoBlocks Difference

Built from the ground up — no legacy constraints, no shortcuts. Every architectural decision is deliberate, documented, and testable.

Security Architecture

Output Escaping

All PHP output uses esc_attr(), esc_html(), esc_url(). Zero raw echoes. Sanitize at input, escape at output.

REST Endpoints

All writes gate on current_user_can(). All SQL via $wpdb->prepare(). SVG through allow-list sanitizer.

CSS Injection Prevention

StyleNormalizer rejects < > ". CSS in generatedCss attribute, never in style="". WP Filesystem for cache writes.
